## 三、功能介绍参考答案

### 1. 你这个个人博客的整体构思是什么？

这个项目的构思是做一个“信息安全学习者的个人安全实验室”，不是单纯的文章展示站。博客负责沉淀文章、学习笔记和项目实践；安全闯关模块把安全知识做成交互式小游戏；数据统计模块观察访问和文章热度；MCP Server 则是对 Agent 管理博客的探索。

### 2. 为什么把它设计成信息安全 / terminal / security lab 风格？

因为博客主题本身包含 Web 安全、CTF、AI 工具和项目实践。terminal 风格更贴合安全学习者和技术笔记的气质，也能避免普通 SaaS 模板那种泛化感，让网站更有个人特色。

### 3. 这个网站主要面向什么使用场景？

主要用于展示个人介绍、发布技术文章、记录项目实践，并作为一个小型全栈项目练习平台。它既是个人博客，也是我学习前后端、数据库、Docker、CI/CD 和云服务器部署的实践载体。

### 4. 你希望别人访问这个博客时先看到什么？

希望访问者先看到我个人的技术方向和博客主题，比如 Web 开发、信息安全、AI 工具、项目实践，以及右侧的安全实验室入口。这能让人快速理解这个网站不是普通生活博客，而是偏技术和安全方向的知识库。

### 5. 你觉得这个项目和普通博客相比有什么自己的特点？

它除了文章、评论、点赞、收藏这些基础博客功能，还加入了安全闯关小游戏、排行榜、数据统计后台、MCP Server 和 GitHub 登录接口。整体 UI 也围绕 security lab 风格设计，更像个人安全实验室。

### 6. 你现在完成了哪些主线功能？

完成了用户注册登录、文章创建和展示、个人介绍、头像展示、评论区、点赞、收藏、文章搜索、数据统计、Docker 部署和 CI/CD。网站可以通过公网 IP 访问，域名备案完成后可以切换到域名访问。

### 7. 哪些功能是你觉得比较有探索性质的？

比较有探索性质的是安全闯关小游戏、数据埋点统计、MCP Server、GitHub 第三方登录和 CI/CD 自动部署。这些功能不只是普通博客必需项，而是对 Agent、安全小游戏和工程化部署的扩展实践。

### 8. 如果再做两周，你最想继续完善哪一块？

我最想继续完善两个方向：第一是安全闯关，增加 URL Inspector、XSS Hunter、SQL Injection Guard 等更多关卡；第二是 AI 自动化，比如 GitHub Trending 自动解读、技术论文 Digest 和文章摘要。

## 四、技术栈问答

### 1. 为什么选择 React 做前端？

React 组件化清晰，适合构建交互式页面，比如文章卡片、评论区、登录状态、排行榜和安全闯关题目都可以拆成组件。React 生态也成熟，适合做个人项目和后续扩展。

### 2. 为什么选择 FastAPI 做后端？

FastAPI 上手快，类型提示清晰，自动生成接口文档，适合快速开发 REST API。它和 Pydantic、SQLAlchemy 配合比较自然，能比较清楚地组织请求数据、响应数据和数据库模型。

### 3. 为什么数据库选择 PostgreSQL？

PostgreSQL 比 SQLite 更适合线上部署和长期扩展，支持更完整的并发、事务和数据类型。后续如果做全文检索、统计分析或向量检索，也更容易扩展。

### 4. 为什么没有只做静态博客？

静态博客适合纯文章展示，但这个项目要求有用户、评论、点赞、收藏、数据统计和小游戏排行榜，这些都需要后端和数据库支持，所以我选择做全栈动态博客。

### 5. React 组件化在这个项目里体现在哪里？

首页、文章卡片、侧边栏、登录页、文章详情、评论区、安全闯关、排行榜等都可以拆成独立组件。这样改 UI 或功能时可以局部修改，不需要重写整个页面。

### 6. FastAPI 的接口文档或类型校验对你有什么帮助？

FastAPI 可以根据路由和 schema 自动生成接口文档，也能用 Pydantic 校验请求体。这样前后端联调时更清楚每个接口需要什么字段、返回什么数据，也能减少手写校验代码。

### 7. 这个技术栈有没有什么不方便的地方？

前后端分离会带来 CORS、认证 token、部署反向代理等额外复杂度。Docker 部署时也需要处理环境变量、数据库连接、端口映射和容器间网络，比单纯静态页面复杂。

### 8. 如果重新选，你还会选 React + FastAPI 吗？

会。这个组合对教学项目比较合适，既能练前端组件化，也能练后端 API、数据库和部署。它的学习成本适中，工程边界也比较清楚。

## 五、项目结构问答

### 1. `frontend` 目录主要放什么？

`frontend` 主要放 React 前端代码，包括页面、组件、样式、API 请求封装和构建配置。用户看到的首页、文章页、登录页、安全闯关页都在前端实现。

### 2. `backend` 目录主要放什么？

`backend` 主要放 FastAPI 后端代码，包括 API 路由、数据库模型、Pydantic schema、认证逻辑、数据库连接、初始化数据和测试代码。

### 3. `nginx` 目录是干什么的？

`nginx` 目录放 Nginx 配置。Nginx 在部署时作为入口服务，负责把普通页面请求转发到前端容器，把 `/api` 请求转发到后端容器。

### 4. `docker-compose.yml` 起什么作用？

`docker-compose.yml` 用来编排多个容器，包括数据库、后端、前端和 Nginx。部署时通过一条 `docker compose up -d --build` 就能构建并启动整套服务。

### 5. `.github/workflows` 里是什么？

里面是 GitHub Actions 工作流配置，包括 CI 检查和自动部署。代码推送到 GitHub 后，Actions 可以自动连接服务器、更新代码、重建容器并做健康检查。

### 6. 后端的 model、schema、api 分别是什么？

model 是数据库模型，对应数据库表；schema 是请求和响应的数据结构，用于校验和序列化；api 是具体接口路由，处理请求、调用数据库并返回结果。

### 7. 前端页面和组件是怎么组织的？

页面负责整体路由和数据加载，组件负责局部 UI，比如文章卡片、侧边栏、按钮、评论区和安全闯关卡片。这样更容易复用和维护。

### 8. 静态资源和上传文件怎么处理？

前端静态资源构建后由前端容器提供，上传文件如头像图片由后端保存到 uploads 目录，并通过 `/uploads` 路径提供访问。线上使用 Docker volume 保证上传文件不会随容器重建丢失。

## 六、前端细节问答

### 1. 前端怎么请求后端 API？

前端通过 HTTP 请求访问 `/api` 开头的接口。浏览器请求先到 Nginx，Nginx 再把 `/api` 转发给后端 FastAPI 服务。

### 2. 登录状态存在什么地方？

登录成功后后端返回 JWT token，前端保存 token，并在后续需要认证的请求中放到 `Authorization: Bearer <token>` 请求头里。

### 3. 文章列表是怎么渲染的？

前端请求文章列表 API，拿到文章数组后遍历渲染为文章列表或文章卡片。每篇文章展示标题、摘要、日期、标签、浏览数、点赞数和评论数等信息。

### 4. 点赞和收藏点击后前端怎么更新？

用户点击点赞或收藏后，前端调用对应 API。请求成功后，前端更新当前文章的状态和数量，让页面即时反馈用户操作。

### 5. 为什么要做响应式设计？

因为验收或实际访问可能在不同屏幕上进行，比如桌面、平板和手机。响应式设计能保证侧边栏、文章列表和右侧栏在不同宽度下仍然可用。

### 6. 移动端和桌面端布局有什么区别？

桌面端通常是主内容加右侧栏的两栏布局，信息密度更高；移动端会把侧栏和内容上下排列，保证文字和按钮不被挤压。

### 7. 这次 UI 为什么避免普通 SaaS 风格？

因为这是个人技术博客，不是商业落地页。过大的渐变 Hero、空旷卡片和厚重阴影会显得模板化，所以改成深蓝 terminal/security lab 风格，更贴合信息安全主题。

### 8. 你怎么保证页面风格统一？

通过统一颜色变量、边框、按钮、卡片、字体层级和间距，让首页、文章页、登录页、安全闯关页都保持深蓝背景、细边框和 cyan 高亮。

## 七、后端细节问答

### 1. 后端有哪些主要 API？

主要包括认证 API、用户信息 API、文章 API、评论 API、点赞收藏 API、访问埋点 API、安全游戏 API、统计 API 和 MCP API。

### 2. 注册和登录流程是什么？

注册时用户提交用户名、邮箱和密码，后端检查是否重复，然后哈希密码并保存用户。登录时后端校验邮箱和密码，成功后返回 JWT token。

### 3. JWT token 是怎么用的？

JWT token 用来表示登录身份。前端在需要认证的请求中带上 token，后端解析 token 获取用户 ID 和角色，再决定是否允许执行操作。

### 4. 怎么判断用户是否有管理员权限？

用户表里有 role 字段。普通用户是 `user`，管理员是 `admin`。后端在创建文章、修改文章、统计后台等敏感接口中检查当前用户角色是否为管理员。

### 5. 评论接口和文章接口有什么关系？

评论属于某篇文章，评论表里会保存 post_id。获取文章详情时可以加载文章的评论，提交评论时也需要指定对应文章。

### 6. 点赞、收藏如何避免重复？

点赞和收藏通常通过用户 ID 和文章 ID 关联。后端在创建记录前会检查该用户是否已经对这篇文章点过赞或收藏过，避免重复计数。

### 7. 数据统计接口怎么统计访问量、评论数、点赞数？

访问量来自埋点事件表，评论数来自评论表，点赞数来自点赞表。后端通过 SQL 聚合查询统计总量、热门文章和排行榜数据。

### 8. MCP 接口为什么要加认证？

因为 MCP 暴露的是给 Agent 使用的博客管理能力。如果不加认证，外部用户可能读取不该读的数据，甚至调用创建、修改、删除文章等操作，所以必须用 Bearer token，并对管理操作检查管理员权限。

### 9. GitHub 第三方登录的大致流程是什么？

前端点击 GitHub 登录后跳到后端 `/auth/github/start`，后端重定向到 GitHub 授权页。GitHub 授权后回调后端 `/auth/github/callback`，后端用 code 换 access token，再读取 GitHub 用户信息，创建或绑定本地用户，最后返回本站 JWT token。

## 八、数据库问答

### 1. 数据库里有哪些主要表？

主要有用户表、文章表、评论表、点赞表、收藏表、个人资料表、访问事件表、安全游戏关卡和分数表等。

### 2. 用户表和文章表是什么关系？

一个用户可以创建多篇文章，文章表中保存 author_id 指向用户表。这是一对多关系。

### 3. 评论表和文章表是什么关系？

一篇文章可以有多条评论，每条评论属于一篇文章，所以评论表通过 post_id 关联文章表。

### 4. 点赞、收藏为什么单独建表？

因为点赞和收藏是用户与文章之间的关系。单独建表可以记录哪个用户对哪篇文章操作过，也方便取消点赞、取消收藏和统计数量。

### 5. 访问统计事件为什么可以单独建 event 表？

访问统计是一类行为日志，未来可能不只记录 page_view，还可以记录搜索、点击、停留等行为。单独建 event 表更灵活，便于后续做数据分析。

### 6. ORM 是什么？项目里怎么用 SQLAlchemy？

ORM 是对象关系映射，可以用 Python 类表示数据库表，用对象操作代替手写 SQL。项目里用 SQLAlchemy 定义模型，并通过 Session 查询、插入和更新数据。

### 7. PostgreSQL 和 SQLite 相比有什么区别？

SQLite 是文件型数据库，适合本地开发和小项目；PostgreSQL 是独立数据库服务，更适合线上部署、多连接访问、事务和长期扩展。

### 8. 如果数据量变大，你会怎么优化？

可以给常用查询字段加索引，比如文章状态、创建时间、用户 ID、post_id；对热门文章和统计数据做缓存；文章搜索可以接入全文检索或专门的搜索服务。

## 九、网络问答

### 1. 用户访问网站时，请求经过了哪些环节？

用户浏览器访问公网 IP 的 80 端口，请求先到阿里云服务器上的 Nginx 容器。Nginx 根据路径把请求转发到前端容器或后端容器，后端再访问数据库容器。

### 2. Nginx 在项目里起什么作用？

Nginx 是统一入口和反向代理。它负责接收外部请求，把页面请求转发给前端，把 `/api` 请求转发给后端，同时可以设置安全响应头和上传大小限制。

### 3. `/api` 请求和普通页面请求分别转发到哪里？

普通页面请求转发到 frontend 容器，`/api` 开头的请求转发到 backend 容器。这样浏览器只需要访问同一个公网地址。

### 4. 公网 IP、域名、安全组分别是什么概念？

公网 IP 是服务器在互联网中的地址；域名是更容易记忆的名字，解析后指向公网 IP；安全组是云服务器的防火墙规则，决定哪些端口允许外部访问。

### 5. 为什么安全组要开放 80 端口？

HTTP 默认使用 80 端口。网站通过 Nginx 暴露在 80 端口，所以阿里云安全组必须允许外部访问 80 端口，否则浏览器无法打开网站。

### 6. 备案期间为什么可以先用公网 IP 访问？

域名备案影响的是域名访问，公网 IP 本身可以直接访问服务器。如果网站还没有绑定备案完成的域名，可以临时用公网 IP 验收和测试。

### 7. CORS 是什么？为什么前后端分离会遇到它？

CORS 是浏览器的跨域资源共享机制。如果前端和后端域名、协议或端口不同，浏览器会限制请求。项目里后端配置允许的来源，解决前后端分离时的跨域问题。

### 8. HTTP 请求从浏览器到后端，大概发生了什么？

浏览器发起 HTTP 请求到服务器公网 IP，Nginx 接收请求并根据路径转发给后端 FastAPI。FastAPI 处理路由、认证和数据库操作，然后返回 JSON，Nginx 再把响应返回给浏览器。

## 十、Docker 与部署问答

### 1. 为什么要用 Docker？

Docker 可以把运行环境打包起来，避免“本机能跑、服务器不能跑”的问题。前端、后端、数据库和 Nginx 都容器化后，迁移部署更稳定。

### 2. Docker Compose 里有哪些服务？

主要有 `db`、`backend`、`frontend`、`nginx` 四个服务。`db` 是 PostgreSQL，`backend` 是 FastAPI，`frontend` 是构建后的 React 应用，`nginx` 是统一入口。

### 3. frontend、backend、db、nginx 分别是什么容器？

frontend 负责提供前端静态页面；backend 负责 API；db 负责保存数据；nginx 负责接收外部请求并反向代理。

### 4. 容器之间怎么通信？

Docker Compose 会创建内部网络，容器可以通过服务名互相访问，比如后端用 `db:5432` 访问 PostgreSQL，Nginx 用 `backend:8000` 访问后端。

### 5. 数据库数据怎么持久化？

数据库使用 Docker volume 保存数据。即使容器重建，volume 里的数据库文件仍然保留，不会随容器删除而丢失。

### 6. `.env` 文件为什么不能直接提交？

`.env` 里通常包含数据库密码、SECRET_KEY、管理员密码、GitHub OAuth Secret 等敏感信息。如果提交到 GitHub，可能导致账号和服务器被攻击。

### 7. 部署时为什么要 build？

因为前端和后端代码更新后，需要重新构建镜像。前端要生成静态资源，后端镜像要包含最新代码和依赖，所以部署时执行 `docker compose up -d --build`。

### 8. 如果服务器重启，服务怎么恢复？

Compose 服务配置了 restart 策略后，Docker 会在服务器重启后自动拉起容器。数据库数据通过 volume 持久化，不会因为重启丢失。

### 9. 你遇到过部署失败吗？怎么排查的？

遇到过。排查思路是先看 GitHub Actions 是否失败，再看服务器上容器是否启动，使用 `docker compose ps`、`docker compose logs`、`curl /api/health` 判断是构建失败、容器失败还是反向代理问题。

### 10. CI/CD 的意义是什么？

CI/CD 可以把测试、构建和部署流程自动化。每次提交代码后自动检查和部署，减少手工操作错误，也能留下清晰的版本记录。

## 十二、安全相关问答

### 1. 你的博客有哪些基础安全措施？

包括密码哈希存储、JWT 登录鉴权、管理员权限检查、敏感配置放在 `.env`、Nginx 安全响应头、MCP 需要 Bearer token、后台接口限制管理员访问。

### 2. 密码是明文存储的吗？

不是。密码会经过哈希处理后再存储，登录时比较的是用户输入密码和哈希结果，不直接保存明文密码。

### 3. JWT 有什么风险？

JWT 如果泄露，攻击者可能冒充用户。所以 token 应该设置过期时间，前端要避免把 token 暴露到不安全环境，后端也要保护 SECRET_KEY。

### 4. 管理员接口如何保护？

后端会先通过 JWT 获取当前用户，再检查用户 role 是否为 admin。只有管理员可以创建、修改、删除文章或访问统计后台等敏感操作。

### 5. GitHub OAuth 为什么需要 callback？

GitHub 授权完成后需要把用户带回网站，这个回来的地址就是 callback。后端在 callback 中用 code 换取 GitHub access token，再获取用户信息并登录本站。

### 6. 为什么 secret key、数据库密码不能写进代码？

这些是敏感信息，如果写进代码并提交到 GitHub，别人可能用它登录数据库、伪造 token 或控制服务。正确做法是放在服务器 `.env` 和 GitHub Secrets 中。

### 7. Nginx 里设置的一些安全 header 有什么作用？

例如 `X-Frame-Options` 可以减少被 iframe 嵌套的风险，`X-Content-Type-Options` 减少 MIME 嗅探风险，`Referrer-Policy` 控制来源信息泄露。

### 8. 你觉得这个项目还有哪些安全不足？

还可以继续完善 HTTPS、限流、防暴力破解、CSRF 防护、日志审计、上传文件类型校验、后台操作审计和更细粒度的权限控制。

## 十三、安全闯关小游戏问答

### 1. 为什么选择做安全小游戏？

因为项目主题是信息安全，安全知识如果只写文章会比较静态。小游戏可以把钓鱼邮件识别、可疑链接判断、XSS 和 SQL 注入等知识变成任务挑战，更有互动性。

### 2. Phishing Detective 的灵感是什么？

灵感来自常见的钓鱼邮件和安全意识训练。很多安全入门都会从识别伪装链接、异常发件人、诱导点击等场景开始，所以很适合作为第一个小游戏关卡。

### 3. 游戏和博客主题有什么关系？

博客沉淀理论和笔记，游戏提供实践和互动。它们都围绕信息安全学习，只是表现方式不同，一个偏内容，一个偏体验。

### 4. 排行榜为什么能增强可玩性？

排行榜能给用户反馈和目标，让答题不只是一次性表单，而像一个可以挑战成绩的小游戏。它也能展示分数、正确率和用时。

### 5. 这个小游戏现在还有哪些限制？

目前题目数量和关卡类型还比较少，主要是选择题形式。后续可以加入更多任务类型，比如 URL Inspector、XSS Hunter 和 SQL Injection Guard。

### 6. 未来可以扩展哪些关卡？

可以扩展 Phishing Detective、URL Inspector、Password Audit、XSS Hunter、SQL Injection Guard 等关卡，分别训练钓鱼识别、链接判断、密码强度、危险输入和 SQL 注入风险识别。

### 7. 为什么你觉得游戏方向适合作为支线探索？

因为它和信息安全主题天然契合，而且能体现自己的创意。相比简单增加浏览历史或点赞历史，安全小游戏是一个不同类型的进阶功能，能展示交互、数据记录和排行榜。

## 十四、MCP / Agent 探索问答

### 1. 为什么想做 MCP Server？

因为 Agent 正在变成新的开发和管理入口。我想尝试把博客能力暴露给 Agent，让它将来可以读取文章、创建文章、更新文章和查看统计数据。

### 2. MCP Server 和普通 API 有什么区别？

普通 API 通常是给前端调用的，接口围绕页面功能设计。MCP 更偏向给 Agent 暴露工具能力，重点是让 Agent 知道有哪些工具、每个工具需要什么参数、调用后返回什么结果。

### 3. Agent 能通过 MCP 做什么？

目前可以读取个人资料、列出文章、读取文章；管理员 token 下可以创建、更新、删除文章，也可以读取统计数据。未来可以扩展自动写摘要、定时生成周报、整理 GitHub Trending 等。

### 4. 为什么 MCP 需要权限控制？

因为 MCP 能力比较强，特别是创建、更新、删除文章这种操作。如果没有权限控制，任何人或任何 Agent 都可能篡改博客内容，所以需要 Bearer token 和管理员角色判断。

### 5. 你觉得 Agent 管理博客有什么想象空间？

未来 Agent 可以帮我整理草稿、生成摘要、推荐标签、分析热门文章、定时抓取技术新闻，甚至根据 GitHub Trending 自动生成学习笔记。

### 6. 这个功能目前完成到什么程度？

目前已经有 `/api/mcp` 接口，能返回 MCP 信息和工具列表，也能通过认证调用博客相关工具。它是一个轻量版本，主要证明博客能力可以被 Agent 化调用。

### 7. 你觉得它的难点在哪里？

难点在权限、安全和工具边界。Agent 能力越强，越需要控制它能做什么、不能做什么；同时还要保证接口返回结构稳定，方便 Agent 正确理解。

域名备案期间可以先用公网 IP 访问网站。等备案完成后，把域名 DNS 解析到服务器公网 IP，并在 Nginx 和 OAuth callback 中切换为域名即可。

当前备案期间主要使用公网 IP 访问。技术上已经具备通过域名访问的条件，后续备案完成后配置 DNS 和相关 callback 地址即可。

### 8. 为什么 GitHub 登录线上可能显示未启用？

因为第三方登录需要在服务器 `.env` 中配置 GitHub OAuth 的 Client ID、Client Secret 和 Redirect URI。代码已经支持，但为了安全，密钥不能写进仓库。

### 9. 为什么 MCP 写操作需要管理员？

因为创建、更新和删除文章会改变博客内容，属于高权限操作。为了避免普通用户或未授权 Agent 修改内容，需要管理员权限。

### 10. 如果服务器上后端挂了，你怎么排查？

先访问 `/api/health` 判断后端是否可用；再看 `docker compose ps` 判断容器状态；然后用 `docker compose logs backend nginx` 看日志；最后根据错误判断是构建、数据库连接、环境变量还是 Nginx 转发问题。
